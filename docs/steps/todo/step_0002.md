Step 0002 - Domini dedicati senza database

Stato attuale

- Non esiste ancora una tabella `domains`.
- I domini dedicati sono configurati via file.
- Rails entra sempre da una sola route:

```ruby
root "dedicated_domains#show"
```

- Ogni dominio puo scegliere una action interna del `DedicatedDomainsController`.
- Le action sono whitelistate nel controller, non chiamate dinamicamente con `public_send`.

Flusso attuale

```text
request.host
-> DedicatedDomainsController#show
-> DedicatedDomain legge config/dedicated_domains.yml
-> action
-> render della pagina
```

Esempio:

```text
posturacorretta.org
-> action: markpostura
-> render "pages/markpostura"
```

File da modificare quando aggiungi un dominio

1. DNS, fuori dal repo

Nel pannello del provider dominio:

```text
nuovodominio.org      A      IP_SERVER
www.nuovodominio.org  CNAME  nuovodominio.org
```

2. Kamal

File:

```text
config/deploy.yml
```

Aggiungere il dominio in:

```yaml
proxy:
  hosts:
    - nuovodominio.org
    - www.nuovodominio.org
```

e anche in:

```yaml
env:
  clear:
    APP_HOSTS: "flowpulse.net,www.flowpulse.net,nuovodominio.org,www.nuovodominio.org"
```

3. Rubrica domini dell'app

File:

```text
config/dedicated_domains.yml
```

Aggiungere:

```yaml
nuovodominio.org:
  action: markpostura
  locale: it
  primary: true

www.nuovodominio.org:
  canonical_host: nuovodominio.org
```

Perche va ripetuto

Per ora il dominio va ripetuto in piu livelli perche ogni livello fa un lavoro diverso:

```text
DNS
-> porta il dominio al server

Kamal proxy.hosts
-> dice al proxy quali host accettare e per quali chiedere SSL

Rails APP_HOSTS
-> autorizza gli Host header in produzione

config/dedicated_domains.yml
-> decide quale action Rails usare per quel dominio
```

Regola `www`

- `www.nome.org` e alias canonico di `nome.org`.
- Nel file YAML, `www.nome.org` usa `canonical_host`.
- Un sottodominio diverso da `www`, per esempio `academy.nome.org`, non e alias automatico.
- In futuro un sottodominio diverso da `www` potra puntare a una action, webapp o experience diversa.

Controller attuale

File:

```text
app/controllers/dedicated_domains_controller.rb
```

Forma:

```ruby
class DedicatedDomainsController < ApplicationController
  allow_unauthenticated_access
  before_action :set_dedicated_domain

  def show
    I18n.with_locale(Current.dedicated_domain&.locale || I18n.default_locale) do
      return redirect_to_canonical_host if Current.dedicated_domain&.canonical_host

      case Current.dedicated_domain&.action
      when "markpostura"
        markpostura
      else
        mvp_home
      end
    end
  end

  private

  def markpostura
    render "pages/markpostura"
  end

  def mvp_home
    render "pages/mvp_home"
  end
end
```

Quando aggiungi una nuova action:

1. Aggiungi il dominio in `config/dedicated_domains.yml`.
2. Aggiungi il ramo nel `case`.
3. Aggiungi il metodo privato che renderizza la pagina.

Development

Per simulare un dominio senza modificare `/etc/hosts`:

```bash
DEDICATED_DOMAIN_HOST_OVERRIDE=posturacorretta.org bin/rails server
```

Poi apri:

```text
http://localhost:3000
```

Nota:

- usare `posturacorretta.org` per vedere direttamente la pagina;
- usare `www.posturacorretta.org` solo se vuoi testare il redirect canonico;
- `DEDICATED_DOMAIN_HOST_OVERRIDE` e solo per development;
- non metterlo in `config/deploy.yml`;
- non usarlo in produzione.

Produzione

Checklist:

1. DNS apex punta all'IP server.
2. DNS `www` e CNAME verso apex.
3. Porte 80 e 443 aperte.
4. `config/deploy.yml` contiene il dominio in `proxy.hosts`.
5. `config/deploy.yml` contiene il dominio in `APP_HOSTS`.
6. `config/dedicated_domains.yml` contiene apex e `www`.

Deploy:

```bash
bin/kamal config cat
bin/kamal deploy
```

Se il proxy non prende i nuovi host:

```bash
bin/kamal proxy reboot
bin/kamal deploy
```

SSL

- `proxy.ssl: true` abilita SSL nel proxy Kamal.
- Kamal proxy puo gestire Let's Encrypt per gli host dichiarati.
- I DNS devono gia puntare al server prima del certificato.
- Le porte 80 e 443 devono essere raggiungibili.

Futuro

Quando ci sara il database:

```text
request.host
-> DedicatedDomainsController#show
-> Domain.find_by(hostname: request.host)
-> target pubblico
-> render pagina/webapp/experience
```

Il `Domain` futuro non dovra puntare al brand generico.
Dovra puntare a un target pubblico:

- pagina/webapp dedicata del brand;
- experience che si comporta come webapp pubblica;
- altro target ancora da definire.

Possibile schema futuro:

```text
Domain
- hostname
- locale
- target_type
- target_id
- target_kind
- primary:boolean
- active:boolean
```

Regola finale

Adesso:

```text
domain YAML -> action -> render
```

Domani:
