Step 0002 - Domini da database

bisogna aggiungere a domain le info che si vedono in head title immagini ecc...

Stato attuale

- `Domain` e la sorgente runtime vera, sia in development che in production.
- `config/dedicated_domains.yml` resta come file di seed iniziale.
- L'export admin scarica `flowpulse_domains.yml`.
- Rails entra sempre da una sola route:

```ruby
root "domains#show"
```

Flusso runtime

```text
request.host
-> DomainsController#show
-> Domain.find_by(hostname: request.host)
-> target_controller/target_action oppure action
-> render
```

Regola di rendering:

```text
se domain.target_controller presente
-> render "#{domain.target_controller}/#{domain.target_action}"

altrimenti
-> usa domain.action (es: markpostura o mvp_home)
```

Per ora le action whitelistate sono:

```text
mvp_home
markpostura
```

Quando aggiungere un dominio

1. DNS, fuori dal repo

Nel provider dominio:

```text
nuovodominio.org      A      IP_SERVER
www.nuovodominio.org  CNAME  nuovodominio.org
```

2. Kamal proxy e Rails host authorization

File:

```text
config/deploy.yml
```

Aggiungere in:

```yaml
proxy:
  hosts:
    - nuovodominio.org
    - www.nuovodominio.org
```

e in:

```yaml
env:
  clear:
    APP_HOSTS: "flowpulse.net,www.flowpulse.net,nuovodominio.org,www.nuovodominio.org"
```

3. Database

Creare il record da admin:

```text
/admin/domains
```

oppure importare un file YAML dalla UI superadmin:

```text
/admin/domains/export
```

4. YAML, se vuoi tenerlo come sorgente/backup

File:

```text
config/dedicated_domains.yml
```

Esempio:

```yaml
nuovodominio.org:
  target_controller: pages
  target_action: markpostura
  locale: it
  primary: true

www.nuovodominio.org:
  canonical_host: nuovodominio.org
```

Poi import:

```text
/admin/domains/export
```

Export dal database

Per esportare i domini dal DB, usare la UI superadmin:

```text
/admin/domains/export
```

La pagina permette di:

- scaricare `flowpulse_domains.yml`;
- caricare/importare un file `.yml` o `.yaml`.

I task rake restano solo come utilita tecnica:

```bash
bin/rails domains:import
bin/rails domains:export
```

Regola `www`

- `www.nome.org` e alias canonico di `nome.org`.
- Nel DB, il record `www.nome.org` usa `canonical_host`.
- Un sottodominio diverso da `www`, per esempio `academy.nome.org`, non e alias automatico.
- In futuro un sottodominio diverso da `www` potra puntare a una action, webapp o experience diversa.

Development

Per simulare un dominio senza modificare `/etc/hosts`:

```bash
DEDICATED_DOMAIN_HOST_OVERRIDE=posturacorretta.org bin/rails server
```

Poi apri:

```text
http://localhost:3000
```

Nota:

- `DEDICATED_DOMAIN_HOST_OVERRIDE` e solo per development;
- non metterlo in `config/deploy.yml`;
- non usarlo in produzione.

Produzione

Checklist:

1. DNS apex punta all'IP server.
2. DNS `www` e CNAME verso apex.
3. Porte 80 e 443 aperte.
4. `config/deploy.yml` contiene il dominio in `proxy.hosts`.
5. `config/deploy.yml` contiene il dominio in `APP_HOSTS`.
6. Il DB contiene il record `Domain`.

Deploy:

```bash
bin/kamal config cat
bin/kamal deploy
```

Dopo la prima migrazione/import:

```bash
bin/kamal app exec "bin/rails db:migrate"
```

Poi importare i domini da:

```text
/admin/domains/export
```

oppure, se serve da shell:

```bash
bin/kamal app exec "bin/rails domains:import"
```

Se il proxy non prende i nuovi host:

```bash
bin/kamal proxy reboot
bin/kamal deploy
```

SSL

- `proxy.ssl: true` abilita SSL nel proxy Kamal.
- Kamal proxy puo gestire Let's Encrypt per gli host dichiarati.
- I DNS devono gia puntare al server prima del certificato.
- Le porte 80 e 443 devono essere raggiungibili.

Futuro

Il `Domain` futuro non dovra puntare al brand generico.
Dovra puntare a un target pubblico:

- pagina/webapp dedicata del brand;
- experience che si comporta come webapp pubblica;
- altro target ancora da definire.

