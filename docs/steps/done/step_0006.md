# Step 0006 - Dashboard app shell unica

## Obiettivo

Rifare la UX delle aree dashboard/admin/workspace con una shell unica e stabile.

Il problema emerso nello step 0005 non e solo l'aside: e la coesistenza di layout diversi:

- griglie Tailwind inline nelle view;
- CSS custom `.dashboard-shell`;
- aside sticky;
- pagine admin con contenuto lungo;
- pagine demo/workspace con shell propria;
- fix `overflow` applicati pagina per pagina.

Risultato: quando il main e lungo, aside e contenuto non hanno due scroll realmente indipendenti.

La soluzione e introdurre una `Dashboard App Shell` condivisa.

## 1. Struttura desiderata

```text
body
└── nav/header
└── dashboard-app-shell
    ├── dashboard-app-aside
    │   ├── header ruolo/mondo
    │   ├── nav scrollabile
    │   └── role switch
    └── dashboard-app-main
        └── contenuto pagina scrollabile
```

Regola:

- il body non deve essere lo scroll principale delle dashboard;
- la shell dashboard ha altezza viewport;
- aside e main hanno due scroll verticali separati;
- su mobile la shell puo tornare a flusso naturale.

## 2. CSS base

CSS desiderato:

```css
.dashboard-app-shell {
  height: calc(100dvh - var(--dashboard-header-height, 73px));
  min-height: 0;
  display: grid;
  grid-template-columns: 280px minmax(0, 1fr);
  overflow: hidden;
}

.dashboard-app-aside {
  min-height: 0;
  overflow: hidden;
  display: flex;
  flex-direction: column;
}

.dashboard-app-aside__nav {
  min-height: 0;
  overflow-y: auto;
}

.dashboard-app-main {
  min-height: 0;
  overflow-y: auto;
}
```

Nota: usare CSS reale, non classi Tailwind arbitrarie, cosi non dipende dal build Tailwind.

## 3. Partial da creare

Creare una partial unica:

```text
app/views/shared/_dashboard_shell.html.erb
```

Responsabilita:

- renderizzare `shared/dashboard_aside`;
- creare il contenitore main scrollabile;
- ricevere `current_section`;
- ricevere il contenuto via block.

Esempio:

```erb
<%= render "shared/dashboard_shell", current_section: :role_map do %>
  ...
<% end %>
```

Stato:

- fatto: creata `app/views/shared/_dashboard_shell.html.erb`;
- fatto: la partial renderizza `shared/dashboard_aside`;
- fatto: il contenuto pagina passa via block.

## 4. Aside

L'aside deve diventare compatibile con la shell:

- niente `sticky`;
- niente altezza calcolata inline;
- classi stabili:
  - `dashboard-app-aside`;
  - `dashboard-app-aside__header`;
  - `dashboard-app-aside__nav`;
  - `dashboard-app-aside__switcher`.

La partial `shared/_dashboard_aside.html.erb` resta responsabile del contenuto, non del layout pagina.

Stato:

- fatto: aggiunte classi `dashboard-app-aside`, `dashboard-app-aside__header`, `dashboard-app-aside__nav`, `dashboard-app-aside__switcher`;
- fatto: rimosso uso di sticky/layout critico dalla partial aside;
- fatto: lo scroll verticale appartiene a `dashboard-app-aside__nav`.

## 5. Main

Il main della shell deve essere l'unico scroll verticale del contenuto centrale.

Regole:

- niente `max-h` sulle tabelle se non serve;
- tabelle possono avere solo `overflow-x-auto`;
- pagine lunghe scorrono nel main;
- header pagina scorre insieme al contenuto, salvo futura scelta sticky.

Stato:

- fatto: `dashboard-app-main` e lo scroll verticale del contenuto centrale;
- fatto: `role_map` non usa piu scroll verticale annidato sulla tabella.

## 7. Cosa rimuovere o evitare

Evitare:

- `lg:sticky` sull'aside dentro dashboard shell;
- `lg:self-start` quando serve altezza piena;
- `max-h-[calc(...)]` Tailwind arbitrario per layout critico;
- scroll verticale annidato su tabelle se il main gia scrolla;
- griglie dashboard duplicate in ogni view.

Rimuovere o non usare piu nelle dashboard migrate:

- `dashboard-split-scroll`;
- patch overflow pagina-specifiche;
- wrapper `grid lg:grid-cols-[280px_minmax(0,1fr)]` nelle view.

Stato:

- fatto per le pagine migrate;
- restano alcune viste demo/home legacy con `.dashboard-shell--with-aside`, da trattare in uno step successivo se diventano dashboard reali.
