# Step 0004 - Semplificazione app, ruoli e FlowTree

## Obiettivo

Semplificare `flowpulse_v_5` prima di montare FlowTree o estrarre `flow_roles` in gemma.

La regola guida e separare bene:

- routing dashboard;
- mondi applicativi;
- menu per ruolo;
- demo sandbox;
- API interna dei ruoli.

## 1. Un solo router dashboard

Tenere `/dashboard` come ingresso unico e farlo smistare sempre con `dashboard_home_path`.

Mappa desiderata:

| Ruolo attivo | Destinazione |
| --- | --- |
| traveler / utente normale | `/dashboard/viaggiatore` |
| creator | `/creator_world` |
| teacher | `/teacher` |
| tutor | `/tutor` |
| professional | `/professional` |
| admin | `/admin/dashboard` |
| superadmin | `/admin/dashboard` |
| demo | `/demo/viaggiatori` |

Stato attuale:

- fatto: `/dashboard` usa `dashboard_home_path`;
- fatto: creator/teacher/tutor/professional hanno dashboard dedicate;
- fatto: `/dashboard/viaggiatore` richiede autenticazione.

Beneficio: nei menu e nelle view non serve decidere ogni volta dove deve andare il link Dashboard.

## 2. Separare bene i mondi

Divisione mentale dell'app:

| Mondo | Responsabilita |
| --- | --- |
| Public | landing, esperienze, pagine pubbliche |
| Workspace | traveler, creator, teacher, tutor, professional |
| Admin | admin operativo e superadmin |
| Demo | sandbox/prototipi read-only |

Regola:

- tutto cio che e pubblico resta in route pubbliche;
- tutto cio che e reale per ruoli operativi sta nel workspace;
- tutto cio che e gestione interna sta in admin;
- tutto cio che e prova/prototipo sta in demo.

## 3. Ridurre condizioni nelle view

Obiettivo: evitare condizioni sparse tipo:

```erb
<% if superadmin_user? || teacher_user? %>
```

Preferire strutture centralizzate:

```ruby
FlowRoles::MenuRegistry.visible_for(...)
```

Direzione futura:

```ruby
FlowRoles.can?(Current.user, :manage, :courses)
```

Stato attuale:

- fatto: aside guidata da `FlowRoles::MenuRegistry`;
- fatto: helper ruolo centralizzati;
- fatto: introdotta mini API `FlowRoles.can?` per le prime regole azione/risorsa.

## 4. Unificare dashboard di ruolo

Le dashboard di ruolo devono essere leggere e non duplicare markup.

Stato attuale:

- fatto: partial comune `app/views/shared/_role_dashboard.html.erb`;
- fatto: i controller ruolo passano configurazione:
  - titolo;
  - heading;
  - intro;
  - cards;
  - current section.

Dashboard coinvolte:

- `/creator_world`;
- `/teacher`;
- `/tutor`;
- `/professional`.

## 5. Rendere demo solo read-only

Demo deve avere una regola globale semplice:

- puo vedere prototipi;
- puo cambiare vista;
- non puo modificare dati veri;
- non puo entrare in admin;
- non puo pubblicare FlowTree.

La regola dovrebbe stare in un concern/helper, non sparsa nelle view.

Stato attuale:

- `demo_access` abilita ingresso demo;
- `demo_mode?` indica modalita UI demo;
- `Demo::BaseController` protegge diverse pagine demo;
- fatto: `/demo/pagine/:slug` eredita da `Demo::BaseController`;
- fatto: `Demo::BaseController` accetta solo richieste read-only `GET`/`HEAD`;
- fatto: admin e risorse bloccano utenti admin quando il ruolo attivo e `demo`, con bypass solo per superadmin.

## 6. Mini API interna FlowRoles

Prima della gemma, portare il layer locale verso una API piu chiara:

```ruby
FlowRoles.roles
FlowRoles.label(:teacher)
FlowRoles.assignable_roles
FlowRoles.dashboard_path_for(user, view_context)
FlowRoles.menu_for(user, active_role)
FlowRoles.can_access_role?(user, :teacher)
```

Quando questa API funziona bene localmente, estrarla in gemma diventa piu semplice.

Stato attuale:

- fatto: `FlowRoles::UserRoles`;
- fatto: `FlowRoles::ControllerHelpers`;
- fatto: `FlowRoles::MenuRegistry`;
- fatto: `FlowRoles::MenuItem`;
- fatto: facade `FlowRoles` con metodi pubblici piccoli:
  - `roles`;
  - `label`;
  - `assignable_roles`;
  - `active_role_for`;
  - `dashboard_path_for`;
  - `menu_for`;
  - `can_access_role?`;
  - `can?`.

## 7. Pulizia nomi

Nomi da mantenere:

| Nome | Significato |
| --- | --- |
| `active_role` | vista/dashboard attiva |
| `RoleAssignment` | ruolo reale globale o contestuale |
| `demo_access` | permesso demo |
| `superadmin` | bypass globale |
| `creator_world` | contesto creator/FlowTree |

Regola:

- evitare di reintrodurre `role` generico su `users`;
- `users.role` e ambiguo rispetto a `active_role` e `role_assignments`;
- i permessi reali devono stare in `RoleAssignment`, `demo_access`, `superadmin`.

## 8. Pagine prototipo sotto demo

Tutto cio che e "sto provando una vista" deve stare sotto `/demo`.

Tutto cio che e reale deve stare in:

- `/creator_world`;
- `/teacher`;
- `/tutor`;
- `/professional`;
- `/admin`.

Da verificare:

- quali viste HTML statiche sono ancora prototipi;
- quali pagine demo devono diventare route reali di workspace;
- quali link pubblici devono restare visibili senza login.
