# Step 0005 - UX aside e menu per ruolo

## Obiettivo

Rendere l'aside piu leggibile e coerente con la nuova architettura ruoli.

Dopo lo step 0004, la logica e piu pulita:

- `/dashboard` e router unico;
- `FlowRoles` espone una mini API;
- `FlowRoles::MenuRegistry` guida le voci;
- demo e piu sandbox/read-only;
- le dashboard di ruolo usano una partial comune.

Ora serve migliorare l'esperienza utente dell'aside senza rimescolare permessi e controller.

## 1. Aside diversa per mondo

L'aside non deve essere sempre generica "Workspace / Dashboard".

Titoli consigliati per ruolo/mondo:

| Active role / mondo | Titolo aside | Sottotitolo |
| --- | --- | --- |
| traveler | Esperienze | Eventi, categorie e brand |
| creator | Creator World | Progetti, format e contenuti |
| teacher | Didattica | Percorsi, corsi e lezioni |
| tutor | Accompagnamento | Persone, follow-up e progressi |
| professional | Professionista | Servizi, abilita e disponibilita |
| demo | Demo sandbox | Prototipi e viste read-only |
| admin | Admin | Strumenti operativi |
| superadmin | Superadmin | Governo app, domini e audit |

Implementazione proposta:

- aggiungere metadati UX a `FlowRoles`, per esempio `FlowRoles.aside_context_for(active_role, admin: false)`;
- oppure aggiungerli a `FlowRoles::MenuRegistry`;
- `_dashboard_aside.html.erb` deve leggere titolo/sottotitolo da quella struttura, non hardcodarli.

Stato:

- fatto: aggiunto `FlowRoles.aside_context_for`;
- fatto: `_dashboard_aside.html.erb` usa header dinamico.

## 2. Gruppi di menu

Il menu non deve restare una lista piatta.

Gruppi consigliati:

- `Workspace`
- `Demo`
- `Admin`

Esempio superadmin:

```text
Workspace
- Viaggiatore
- Creator
- Teacher
- Tutor
- Professionista

Demo
- Demo
- Lavoro
- Salute

Admin
- Dashboard Admin
- Risorse
- Domini
- Elenco pagine
- Role map
- Weekplan
```

Regole:

- utente normale vede solo il gruppo utile;
- demo vede solo demo/sandbox e public link ammessi;
- admin vede admin operativo;
- superadmin vede tutti i gruppi;
- locked item quasi mai nell'aside.

Stato:

- fatto: `FlowRoles::MenuItem` ha `group`;
- fatto: `FlowRoles.grouped_menu_for` raggruppa le voci;
- fatto: aside renderizza gruppi con label.

## 3. Separare selettore ruolo dal menu

Il blocco "Mostra dashboard" non deve sembrare una voce di menu.

Possibili soluzioni:

1. Tenerlo nell'aside ma come pannello compatto separato:
   - label `Vista attiva`;
   - badge ruolo;
   - select solo se `Current.user.can_switch_roles?`.
2. Spostarlo solo nel dropdown profilo:
   - aside mostra solo badge ruolo;
   - meno duplicazione.

Decisione consigliata per MVP:

- lasciare il select nell'aside;
- renderlo come blocco secondario;
- tenere lo stesso select anche nel dropdown profilo finche e utile nello sviluppo.

Stato:

- fatto: role switch spostato in pannello `Vista attiva` separato dal nav.

## 4. Stato attivo piu leggibile

La voce attiva deve essere leggibile in tutti i temi.

Regole visuali:

- testo scuro, non bianco su sfondo chiaro;
- bordo sinistro o ring accentato;
- sfondo chiaro;
- subtitle leggibile;
- opzionale badge `Attiva`.

Classi indicative:

```text
active: border-slate-900 bg-slate-100 text-slate-950
open: border-slate-200 bg-white text-slate-900 hover:bg-slate-50
locked: non mostrare, salvo debug/role map
```

## 5. Icone o pill compatte

Per migliorare lo scan senza introdurre una libreria nuova:

| Voce | Pill |
| --- | --- |
| Traveler | EXP |
| Creator | CR |
| Teacher | EDU |
| Tutor | SUP |
| Professional | PRO |
| Demo | LAB |
| Admin | ADM |
| Superadmin | SYS |

Implementazione:

- aggiungere `badge` o `icon_label` a `FlowRoles::MenuItem`;
- renderizzare una pill quadrata/piccola prima del testo;
- niente SVG custom per ora.

Stato:

- fatto: `FlowRoles::MenuItem` ha `badge`;
- fatto: aside renderizza pill compatte.

## 6. Locked item

Nell'aside gli item bloccati confondono piu di quanto aiutino.

Regola:

- non mostrare item non accessibili;
- superadmin vede tutto;
- role map/audit resta il posto dove vedere "tutto per ruolo";
- se serve debug futuro, aggiungere un flag esplicito tipo `show_locked: true`.

## 7. Mobile

Per ora non serve un drawer completo, ma l'aside deve restare leggibile.

Prima soluzione:

- su mobile resta sopra al contenuto;
- header aside compatto;
- gruppi separati con label piccole;
- select ruolo largo 100%.

Da valutare dopo:

- collapse/drawer mobile;
- pulsante `Menu`;
- persistenza stato aperto/chiuso.

## 8. Implementazione proposta

1. Estendere `FlowRoles::MenuItem` con:
   - `group`;
   - `badge`.
   - fatto.
2. Aggiornare `FlowRoles::MenuRegistry.items` assegnando gruppo e badge.
   - fatto.
3. Aggiungere metodo `FlowRoles.grouped_menu_for(user, active_role:, admin: false)`.
   - fatto.
4. Aggiungere metodo `FlowRoles.aside_context_for(active_role, admin: false)`.
   - fatto.
5. Aggiornare `NavigationHelper#dashboard_aside_menu_items` o aggiungere helper dedicato per gruppi.
   - fatto: aggiunto `dashboard_aside_menu_groups`, `dashboard_aside_context`, `dashboard_aside_group_label`.
6. Riscrivere `_dashboard_aside.html.erb` per:
   - header dinamico;
   - gruppi;
   - pill badge;
   - item attivo leggibile;
   - role switch separato.
   - fatto.
7. Aggiornare test menu registry/helper.
   - fatto: test su metadati UX, gruppi e contesto aside.
8. Verificare:
   - traveler;
   - demo;
   - creator/teacher/tutor/professional;
   - admin;
   - superadmin.
   - fatto con test mirati; resta utile una verifica visiva in browser.

## Criteri di completamento

- aside non mostra piu una lista piatta;
- aside mostra titolo coerente con ruolo/mondo;
- menu raggruppato per mondo;
- superadmin vede tutti i gruppi;
- admin non vede workspace/demo non necessari quando e in area admin;
- demo non vede admin;
- item attivo leggibile;
- role map resta audit globale;
- test verdi.
