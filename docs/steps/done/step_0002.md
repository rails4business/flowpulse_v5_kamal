# Step 0002 - Domini da database

## Stato attuale

I domini pubblici sono gestiti da database tramite il model `Domain`.

La route di ingresso pubblica e unica:

```ruby
root "domains#show"
```

Il controller reale e:

```text
app/controllers/domains_controller.rb
```

Non esiste piu una separazione `DedicatedDomain` / `DedicatedDomainsController`.

## Flusso runtime

```text
request.host
-> ApplicationController#dedicated_domain_host
-> DomainsController#show
-> Domain.find_for_host(host)
-> canonical_host redirect, se presente
-> render target_controller/target_action
```

Regola di rendering attuale:

```text
se Domain.target_controller presente
-> render "#{target_controller}/#{target_action}"

altrimenti
-> render "pages/flowpulse"
```

## Regola `www`

- `www.nome.org` e alias canonico di `nome.org`.
- Nel DB, il record `www.nome.org` usa `canonical_host`.
- Un sottodominio diverso da `www`, per esempio `academy.nome.org`, non e alias automatico.
- In futuro un sottodominio diverso da `www` potra puntare a una webapp o experience diversa.

## Admin

L'admin domini passa da `Admin::BaseController`, quindi e accessibile solo ai superadmin.

URL:

```text
/admin/domains
/admin/domains/export
```

La pagina export permette di:

- scaricare `flowpulse_domains.yml`;
- caricare/importare un file `.yml` o `.yaml`.

I task rake restano utilita tecnica:

```bash
bin/rails domains:import
bin/rails domains:export
```

## Aggiungere un dominio

1. DNS, fuori dal repo:

```text
nuovodominio.org      A      IP_SERVER
www.nuovodominio.org  CNAME  nuovodominio.org
```

2. Kamal/Rails host authorization:

File:

```text
config/deploy.yml
```

Aggiungere il dominio in:

```yaml
proxy:
  hosts:
    - nuovodominio.org
    - www.nuovodominio.org
```

e in `APP_HOSTS`.

3. Database:

Creare il record da admin oppure importarlo da YAML:

```yaml
nuovodominio.org:
  target_controller: pages
  target_action: markpostura
  locale: it
  primary: true

www.nuovodominio.org:
  canonical_host: nuovodominio.org
```

## Development

Per simulare un dominio senza modificare `/etc/hosts`:

```bash
DEDICATED_DOMAIN_HOST_OVERRIDE=posturacorretta.org bin/rails server
```

Poi aprire:

```text
http://localhost:3000
```

Nota:

- `DEDICATED_DOMAIN_HOST_OVERRIDE` e solo per development;
- non metterlo in `config/deploy.yml`;
- non usarlo in produzione.

## Produzione

Checklist:

1. DNS apex punta all'IP server.
2. DNS `www` e CNAME verso apex.
3. Porte 80 e 443 aperte.
4. `config/deploy.yml` contiene il dominio in `proxy.hosts`.
5. `config/deploy.yml` contiene il dominio in `APP_HOSTS`.
6. Il DB contiene il record `Domain`.

Deploy:

```bash
bin/kamal config cat
bin/kamal deploy
```

Dopo deploy/migrazione:

```bash
bin/kamal app exec "bin/rails db:migrate"
bin/kamal app exec "bin/rails domains:import"
```

Se il proxy non prende i nuovi host:

```bash
bin/kamal proxy reboot
bin/kamal deploy
```

## Futuro

Il `Domain` futuro non dovra puntare per forza a `target_controller/target_action`.
Potra puntare a un target pubblico piu strutturato:

- pagina/webapp dedicata del brand;
- experience che si comporta come webapp pubblica;
- altro target ancora da definire.

Possibile schema futuro:

```text
Domain
- hostname
- locale
- target_type
- target_id
- primary:boolean
- active:boolean
```

Regola attuale da non rompere:

```text
Domain DB -> target_controller/target_action -> render
```

YAML:

```text
seed/import/export, non sorgente runtime primaria
```
