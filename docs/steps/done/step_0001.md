Impostare deploy produzione con Kamal su Hetzner

Obiettivo

- Preparare Flowpulse per deploy in produzione senza Hatchbox.
- Tenere una sola app Rails.
- Usare Kamal con proxy SSL built-in.
- Partire con un server Hetzner singolo.
- Lasciare il setup pronto per piu domini/brand piu avanti.

Scelta iniziale consigliata

- Hetzner Cloud
- Kamal
- proxy SSL built-in di Kamal
- Let's Encrypt
- GitHub Container Registry

Perche questa scelta

- E piu semplice di gestire Traefik manualmente da subito.
- Riduce i punti di rottura nel primo deploy.
- Lascia comunque una strada aperta a Traefik esplicito piu avanti.

Stato attuale nel repo

- `Dockerfile` presente
- `bin/kamal` presente
- `config/deploy.yml` presente ma ancora da scaffold
- `config/deploy.yml.production-template` presente
- `config/environments/production.rb` gia compatibile con proxy SSL:
  - `config.assume_ssl = true`
  - `config.force_ssl = true`

Valori da decidere prima del deploy vero

- IP server Hetzner
- dominio principale
- username GitHub per GHCR
- token GHCR
- `RAILS_MASTER_KEY`

Template da usare

- `config/deploy.yml.production-template`
- `.kamal/secrets.production-template`

Primo setup consigliato

1. Creare server Hetzner Ubuntu LTS
2. Installare Docker sul server
3. Puntare il dominio al server
4. Copiare `config/deploy.yml.production-template` in `config/deploy.yml`
5. Riempire host, IP e registry
6. Copiare `.kamal/secrets.production-template` in `.kamal/secrets`
7. Inserire i valori reali
8. Eseguire:
   - `bin/kamal setup`
   - `bin/kamal deploy`

Config minima desiderata

- un solo server `web`
- SSL attivo
- volume persistente per `/rails/storage`
- registry GHCR
- `RAILS_MASTER_KEY` come secret

Note architetturali

- Va bene partire con SQLite solo se il volume persistente e i backup sono chiari.
- Se il progetto cresce, il primo salto ragionevole e Postgres managed o accessorio dedicato.
- Per ora il deploy e per app singola, non per brand separati.
- I domini multipli si possono aggiungere dopo nella logica app/domain, non nel deploy iniziale.

Secondo step dopo primo deploy riuscito

- backup storage e database
- host authorization in produzione
- monitoraggio base
- rollback testato
- eventuale separazione job/web
